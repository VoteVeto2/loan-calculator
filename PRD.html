Product Requirements · v1.0

Loan Calculator

A single-page loan tracker for principal, base interest, and custom penalty-rate windows, with running allocation of partial repayments to interest and principal under Civil Code Art. 561 order.

1. Goal

Give a lender or borrower an interactive worksheet that:

  1. Accepts a loan: principal, start date, maturity date, base annual rate.
  2. Accepts zero-or-more penalty windows — intervals with a custom annual rate (典型用途: 罚息利率 for the overdue period or an agreed step-up).
  3. Accepts zero-or-more repayment events — a date and an amount.
  4. Computes, at every event, how much of each repayment went to interest (利息) vs. principal (本金), and the running remainders.
  5. Renders a clean schedule and a totals summary, in Anthropic's warm palette.

2. Non-goals

  • No compounding interest (no 计入本金 of unpaid interest). Simple interest by day count.
  • No multi-currency, FX, or amortization schedule generation (equal-payment 等额本息 / 等额本金).
  • No persistence beyond the browser session (no backend, no login).
  • No statutory cap enforcement (LPR × 4 etc.) — surfaced as an informational warning only.

3. Visual language

Palette is drawn from Anthropic's brand system — Pampas cream paper, Crail orange as the structural accent, Book cloth tan for secondary surfaces, Cloudy gray for hairlines and metadata, Dark for primary text. Type is Poppins (geometric sans) for UI and headings, Lora (humanist serif) for figures and prose.

4. Data model

Loan {
  principal:   number   // CNY, > 0
  startDate:   ISO date // YYYY-MM-DD
  endDate:     ISO date // maturity; must be > startDate
  baseRate:    number   // annual, decimal (0.06 = 6%)
  dayBasis:    365 | 360
  asOf:        ISO date // calc-as-of date; default = today
}

PenaltyWindow {
  id:          string
  start:       ISO date
  end:         ISO date
  rate:        number   // annual, decimal
  // semantics: any time inside [start, end) overrides baseRate
}

Repayment {
  id:          string
  date:        ISO date // must be >= loan.startDate
  amount:      number   // > 0, CNY
}

5. Computation rules

5.1 Day count

Days between two ISO dates = floor((d2 − d1) / 86,400,000), using UTC midnight. Half-open: the start date is counted, the end date is not (so 2025-01-01 to 2025-12-31 = 364 days). User-toggleable basis: 365 (default, 民间借贷 convention) or 360 (金融机构 convention per the PBOC notice).

5.2 Daily rate

dailyRate(d) = applicableAnnualRate(d) / dayBasis, where applicableAnnualRate(d) is the rate of the penalty window covering d, otherwise baseRate. If multiple penalty windows overlap, the one declared later wins.

5.3 Accrual between events

The timeline is split at every "event boundary": startDate, every penalty-window start/end, every repayment date, and asOf. Within each segment, simple interest accrues:

segmentInterest = principalAtSegmentStart * dailyRate(segment) * segmentDays

Accrued unpaid interest carries forward; it is NOT capitalized into principal.

5.4 Repayment allocation (Civil Code Art. 561)

On a repayment date, after accruing interest up to that date, allocate the payment in order:

  1. Interest first — apply payment to outstanding accrued interest; if payment ≥ accrued interest, accrued interest goes to 0 and we move on.
  2. Principal next — remaining payment reduces outstanding principal.
  3. Overage — if payment exceeds principal + accrued interest, the excess is shown as an "overpayment" credit; the loan is closed.

5.5 Totals

  • totalInterestPaid = sum of the interest portions of every repayment.
  • totalPrincipalPaid = sum of the principal portions of every repayment.
  • outstandingPrincipal = principal − totalPrincipalPaid.
  • accruedUnpaidInterest = interest accrued from last event to asOf minus interest already paid.
  • totalOwedAsOf = outstandingPrincipal + accruedUnpaidInterest.
Rounding policy. All arithmetic is carried in IEEE-754 doubles to full precision. Display values are rounded to 2 decimals using banker's rounding only at render time, so successive operations do not accumulate rounding error.

6. UI

6.1 Layout

  1. Header — title + tagline, Crail accent bar.
  2. Loan setup card — 4 inputs (principal, start, end, annual rate) + day-basis toggle + as-of date.
  3. Penalty windows card — table of windows with "+ Add window" button.
  4. Repayments card — table of repayments with "+ Add repayment" button.
  5. Schedule — chronological event ledger: every accrual segment and every repayment, with running balances.
  6. Summary — four big numbers: outstanding principal, accrued unpaid interest, total interest paid, total principal paid.

6.2 Interaction

  • Every input change triggers an immediate recompute (no "Calculate" button).
  • Invalid dates / negative amounts are flagged inline; they do not block recompute of the rest.
  • Rows are removable; the schedule rebuilds.
  • A "Load example" link populates Test Case 2 (the most illustrative).

7. Edge cases

  • Repayment before loan start — ignored, flagged red.
  • Repayment after as-of date — accepted but only counted if its date ≤ as-of.
  • As-of before some repayments — those repayments are not yet applied.
  • Overpayment — excess is shown explicitly; no negative balances rendered.
  • Penalty window outside loan period — clipped to [startDate, asOf].
  • Same-day repayment and penalty boundary — penalty boundary applied first (interest accrues up to repayment), then payment.

9. Implementation

No build step and no runtime dependencies. Classic <script> tags (not ES modules) so the app opens by double-click over file://, where module imports would be blocked by CORS. Concerns are split across small files:

  • index.html — markup only.
  • styles.css — the full palette and layout.
  • js/engine.js — pure computation and formatting (UMD; runs unchanged in Node for testing).
  • js/fixtures.js — the 5 acceptance scenarios + expected figures (UMD; shared by the browser and Node test runners).
  • js/store.js — session persistence over localStorage, with an in-memory fallback when storage is blocked.
  • js/render.js — view layer; pure functions that take data and write DOM with data-* hooks.
  • js/app.js — controller: owns the active session's state, wires events, debounced autosave, self-test runner, boot.

Sessions. Each calculation is a named session { id, name, createdAt, updatedAt, loan, penalties, repayments } stored under localStorage["loanCalc.sessions.v1"]; the active id under loanCalc.activeSession.v1. On load the app resumes the active session (or the most recent, or seeds an example on first run). Edits autosave after a 400 ms debounce. Users can switch, rename, create, duplicate, and delete sessions; at least one always remains.

Testing. A Run self-tests button executes the 5 acceptance cases in the browser. test-engine.mjs runs the same cases headlessly in Node, and test-dom.mjs (jsdom) drives the real UI: edits, autosave, session CRUD, cross-session isolation, and persistence across a simulated reload.

PRD v1.0 — palette and conventions sourced from public Anthropic brand documentation, PRC Civil Code Art. 561, and 最高人民法院民间借贷司法解释 (LPR-based interest cap framework).
